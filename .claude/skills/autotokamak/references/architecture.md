# Architecture

`autotokamak` is a two-layer package. Understanding the layer boundary is essential to avoid making changes in the wrong place.

## Layers

```
autotokamak/
├── src/autotokamak/core/           Layer 1 — physics + IO + schema + logging
├── src/autotokamak/agent/          Layer 2 — URSA runners + prompts + orchestrator
│   ├── prompts/
│   ├── runners/
│   ├── orchestrator/
│   └── dspy/
├── src/autotokamak/data/           dataset schemas
├── src/autotokamak/eval/           metrics, k-fold, diagnostics
├── src/autotokamak/surrogate/      model zoo, AutoML, prediction
├── examples/                       runnable-without-LLM workspaces
├── experiments/                    run traces + reports (gitignored)
└── tools/                          out-of-band scripts (pipeline, probes, reports)
```

## Layer 1 — `autotokamak.core`

The stable physics/IO substrate. Six modules:

- `schema.py` — Pydantic v2 config models (`EquilibriumConfig`, `SweepConfig`, `InvertConfig`).
- `geometry.py` — LCFS + mesh construction (`build_lcfs`, `build_mesh`, `build_mesh_from_config`).
- `solver.py` — TokaMaker solve with isoflux-fallback retry; `OFT_env` singleton.
- `diagnostics.py` — scalar extraction (`R_axis`, `q_95`, etc.), tolerant to OFT API drift.
- `io.py` — atomic file writes, `unified_output_dir`, `utc_run_id`.
- `logging.py` — libc-flushed console logger so OFT compiled output stays in order.

Everything above depends on this layer. Do not duplicate any of these functions in Layer 2, in examples, or in Skill scripts.

## Layer 2 — `autotokamak.agent`

Agent orchestration. Split into:

- `prompts/*.yaml` — the five task descriptions consumed by the runners.
- `runners/plan_execute.py`, `plan_execute_feedback.py`, `meta_loop.py` — CLI entry points.
- `runners/config.py`, `runners/scoring.py`, `runners/trace.py` — shared utilities.
- `orchestrator/actions.py`, `orchestrator/schema.py` — `MetaState`, `dispatch(ActionDecision, state)`.
- `dspy/signatures.py`, `module.py`, `optimize_meta.py`, `trace_loader.py`, `metric_adapter.py`, `metric_surrogate.py`, `metric.py` — DSPy integration + scorers.

Agents generate examples; humans generate agents' prompts and runners.

## Data-flow (Phase 1 → 2 → 3)

```
prompts/dataset_generation.yaml
  ├─→ plan_execute_feedback ─→ examples/dataset_generation/outputs/dataset.h5
                                             │
prompts/surrogate_automl.yaml                │
  ├─→ plan_execute_feedback ─→ examples/surrogate_automl/outputs/winner.pkl
                                             │
prompts/surrogate_meta.yaml                  │
  └─→ meta_loop ─→ examples/surrogate_meta/  │
        │   picks action per iteration:      │
        │   - regen_dataset(overrides) ──────┤ re-runs Phase 1
        │   - extend_search(focus) ──────────┤ runs a fresh Phase 2
        │   - terminate                      │
        └─→ workspace/report.json, winner.pkl, meta_trace.json
```

Every run also writes `experiments/<run_id>/trace.json` for the trace-to-HTML report.

## OFT singleton constraint

`OpenFUSIONToolkit` allows **one `OFT_env` per Python kernel**. All Layer-1 code goes through `autotokamak.core.solver.get_oft_env()` which caches. Layer-2 runners inherit this constraint. Any parallelism over solves must be process-level (`subprocess.Popen`, `multiprocessing` with `spawn`). See `references/core-api.md` and `references/debugging.md`.

## Where to make changes

| Change type | Right place |
|---|---|
| Physics API (mesh, solver, diagnostics) | `src/autotokamak/core/` |
| New config field | `src/autotokamak/core/schema.py` + prompt YAML |
| Agent behavior | prompt YAML in `src/autotokamak/agent/prompts/`, or `runners/`, or `orchestrator/` |
| Scoring / metric | `src/autotokamak/agent/dspy/metric_*.py` |
| New example workspace | `examples/<name>/` (or generated by an agent run) |
| Ad-hoc pipeline glue | `tools/` |

Never: edit `experiments/**` or files inside a generated `examples/<agent-produced>/`. Those are output.
